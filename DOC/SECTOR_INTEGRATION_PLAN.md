# خطة التكامل الذكيّ للقطاعات (حيّة) — القطاع مكوّنٌ عابر في الإضافة · العرض · الوصول · البحث

> **الهدف:** جعل **نطاق النشاطات/القطاعات** (`sectors.ts` — 21 قطاعًا) **مكوّنًا عابرًا (cross-cutting)** حقيقيًّا في كل أسطح المنصّة: كمُدخل عند الإضافة، ومخرَج عند العرض، وخيار عند الوصول/الفلترة، ومُرجِّح عند البحث والتوصية — بحيث يعود **سياق قطاعات المستخدم** ليخصّص تجربته في كل مكان.
> **المرجع:** `DOC/TAXONOMY_PLAN.md` (نموذج التصنيف) + هذه الخطة (طبقة الاستهلاك والتخصيص).

---

## 1. المشكلة (تدقيق مُثبَت)

التصنيف مُدمج **جزئيًّا**: القطاع مكوّن في الإضافة/العرض/البحث عبر عمل B1–B4، **لكن تفضيل المستخدم لا يصل إليه** — الحلقة مقطوعة.

| البُعد | القطاع مكوّن؟ | تفضيل المستخدم يصل؟ | الفجوة |
|---|---|---|---|
| الإضافة (`CreateOpportunity`/`Profile`) | ✅ | ❌ | لا يُبذَر القطاع الافتراضيّ من اهتمام المستخدم |
| العرض/الفلترة (`Opportunities`/`Requests`/`Interviewers`) | ✅ `TaxonomyTree` | ❌ | `treeSel = ref({})` يبدأ فارغًا |
| المطابقة (`matching.ts`/`matchProfile.ts`) | ✅ قطاع↔قطاع | ❌ | ملف الباحث يشتقّ القطاع من المهارات فقط، لا من `interestedSectors` |
| البحث (`globalSearch`) | ✅ واعٍ بالتصنيف | ❌ | `category` وسيط يدويّ لا يُبذَر ولا يُرجَّح بقطاعات المستخدم |
| التوصية (`Dashboard.recommended`) | 🟡 `matchRate` ثابت | ❌ | لا يقرأ `persona.interestedSectors` |

**الدليل القاطع:** `usePersonaStore` يُقرأ في موضعين فقط (`OnboardingPage`+`ProfilePage`)؛ و`interestedSectors` لا يُستهلَك خارج onboarding — **بيانات ميتة**.

## 2. المبدأ المعماريّ: طبقة سياق واحدة (لا ترقيع لكل صفحة)

**القاعدة الذهبية:** لا نبعثر منطق «قطاعات المستخدم» في كل صفحة. نبني **مصدر حقيقة واحدًا** يستهلكه الجميع:

### `useSectorContext()` — composable مركزيّ (جديد: `src/composables/useSectorContext.ts`)
يحلّ **«قطاعات المستخدم الفعّالة»** بترتيب أولويّة، ويعطي أدوات موحّدة:
```ts
interface SectorContext {
  explicit: string[]        // من persona.interestedSectors (اختيار صريح في onboarding/الإعدادات)
  derived: string[]         // مشتقّة من مهارات/خبرات الملف عبر dominantSector (احتياط)
  effective: string[]       // explicit ∪ derived (المصدر الموحّد للتخصيص)
  primary?: string          // القطاع الأبرز (لأغراض الافتراض المفرد)
  has: boolean              // هل لدى المستخدم سياق قطاعيّ أصلًا؟
  // أدوات الاستهلاك الموحّدة:
  seedTree(): { category?: string }         // بذرة فلتر TaxonomyTree (primary، قابل للتجاوز)
  boost(itemSector?: string): number        // معامل ترجيح (0..1) لعنصر حسب قربه من سياق المستخدم
  rankSearch<T>(items, getSector): T[]       // إعادة ترتيب نتائج البحث بالسياق
  matchInput(): { sector?: string }          // مُدخل seekerMatchProfile من السياق لا المهارات فقط
}
```
- يقرأ `PersonaStore.interestedSectors` (المُزامَن مع Laravel) + `ProfileStore` (المهارات) — **يتفاعل** مع تغيّرهما.
- **يحترم الدور النشط والـkind** (§5): الباحث يرى «فرصًا في قطاعاتي»، الجهة ترى «مرشّحين/طلبات في قطاعاتي»، الخبير «تكليفات في قطاعاتي» — نفس السياق، عدسة مختلفة.

## 4. المراحل (مرقّمة، محكمة)

### المرحلة 1 — الأساس (طبقة السياق)
- بناء `useSectorContext()` + دوال `seedTree/boost/rankSearch/matchInput`.
- إضافة تحرير `interestedSectors` في **مركز الإعدادات** (لا يبقى حبيس onboarding).
- اختبارات وحدة للـcomposable (explicit/derived/effective/boost/ترتيب).

### المرحلة 2 — الوصول والعرض (الأسواق الثلاثة)
- بذر `treeSel` من `seedTree()` في `Opportunities`/`Requests`/`Interviewers` **مع تجاوز صريح**: شريحة تبديل «قطاعاتي ⟷ الكل» + زر إعادة تعيين قائم. المستخدم يملك القرار دائمًا (بذر ≠ قفل).
- تمييز/ترتيب عناصر قطاعات المستخدم أولًا (`boost`).
- حالة فارغة (بلا سياق) → لا بذر، سلوك اليوم.

### المرحلة 3 — التوصية والمطابقة
- `Dashboard.recommended` + «موصى به لك»: ترجيح بـ`boost(itemSector)` بدل `matchRate` الثابت.
- `matchProfile.seekerMatchProfile`: `sector` من `matchInput()` (السياق) بدل `dominantSector(skills)` وحدها — مع بقاء الاشتقاق احتياطًا.

### المرحلة 4 — البحث
- `globalSearch.search`: بذر `category` من السياق (قابل للإلغاء) + `rankSearch` يرفع نتائج قطاعات المستخدم + شريحة «ضمن قطاعاتي».

### المرحلة 5 — الإضافة (المُدخلات)
- `CreateOpportunity`: اقتراح القطاع الافتراضيّ من السياق (يدمج مع `autoClassify` القائم — لا يتصادم).
- `Profile`/تفضيلات الباحث: بذر القطاع من السياق.

### المرحلة 6 — الاتّساق والتحقّق
- تدقيق: لا يُقرأ `interestedSectors` مباشرةً في أي صفحة — الكلّ عبر `useSectorContext` (منع التبعثر).
- تحقّق حيّ لكل سطح + اختبارات وحدة + typecheck + الحفاظ على 232 اختبارًا خضراء.
- تأكيد المزامنة الخلفيّة (`persona` في Laravel — منجز).

## 5. قواعد الاتّساق (البزنس واللوجك) — ملزمة

1. **البذر ليس قفلًا:** أي فلتر مبذور من السياق **قابل للتجاوز فورًا** (شريحة «الكل» + إعادة تعيين). المستخدم سيّد تجربته.
2. **متعدّد القطاعات:** `interestedSectors` قائمة — نتعامل بالاتّحاد (union) لا قطاعًا واحدًا؛ `primary` للافتراض المفرد فقط.
3. **العدسة حسب الدور النشط:** نفس السياق، مخرَج مختلف — باحث→فرص، جهة→مرشّحون/طلبات، خبير→تكليفات، مُزكٍّ→زملاء في قطاعاتي. `useSectorContext` يقرأ `authStore.role`.
4. **الـkind:** منشأة قد تُرجَّح لعدسة التوظيف افتراضيًّا، وفرد لعدسة البحث — دون حجب (سمة لا قيد).
5. **سلّم الاحتياط:** explicit → derived (مهارات) → بلا سياق (سلوك اليوم). لا فراغ يكسر الصفحة.
6. **حوكمة التصنيف:** احترام `GENERIC_BLOCKLIST` و`isGenericLabel` وإخفاء S21 (مطبّقة في `sectors.ts`) — لا نعيد فتحها.
7. **المزامنة:** كل تغيير على `interestedSectors` يُزامَن مع Laravel عبر `PersonaStore` (منجز) — فالسياق متّسق عبر الأجهزة.
8. **الأداء:** `useSectorContext` computed خفيف؛ لا استعلامات ثقيلة في كل عنصر — الترجيح دالة نقيّة.
